# SkywalkerGameServer 网络通信架构方案（适配当前框架）

## 1. 目标与约束

本文给出一版可落地、可迭代、适配当前 Skywalker 插件化框架的网络通信方案，目标如下：

- 支持当前 `SSFPlugin_Network` 的服务端/客户端模块继续工作。
- 优先保证游戏场景中的实时性、稳定性和可扩展性。
- 从当前工程状态平滑演进，不做一次性大重构。
- 兼顾后续功能：心跳、重连、消息分发、版本兼容、反作弊基础能力。

当前已知约束：

- 项目采用 C++20 + 插件化架构（Plugin/Module/Object）。
- 现有网络实现以 Socket 为基础，服务端已有 `SSFModule_NetworkServer`，客户端已有 `SSFModule_NetworkClient`。
- 运行流程是框架驱动 `Init/Awake/Start/Tick/Stop/Sleep/Destroy` 生命周期。
- 日志、错误码、脚本配置（`.skywalkerC`）均已存在基础体系。

---

## 2. 推荐方案总览（先做可用，再做高性能）

### 2.1 总体建议：混合式（命令驱动 + 关键状态同步）

在当前框架下，建议采用：

- **上行（客户端 -> 服务器）**：命令驱动（输入/操作命令）
- **下行（服务器 -> 客户端）**：事件 + 增量状态（必要时关键帧快照）
- **传输层先用 TCP**（复用现有实现，快速稳定落地）
- 后续实时要求更高时，再扩展 `UDP/KCP` 通道

这样可以在开发成本和游戏体验之间取得较好的平衡。

### 2.2 为什么适配当前框架

- `SSFPlugin_Network` 天然可以继续作为通信基础插件。
- `SSFModule_NetworkServer/Client` 已具备生命周期与 Tick 机制，适合承载消息收发循环。
- 可以新增子模块，不破坏已有插件边界：
  - 协议编解码模块
  - 消息分发模块
  - 会话管理模块
  - 心跳与重连模块

## 3. 架构分层设计

建议在现有 Network 插件内部形成如下逻辑层：

1) 连接层（Connection Layer）
- 职责：Socket 建连、断连、读写、基础错误处理。
- 现有映射：`SSFObject_ServerSocket`、`SSFObject_ClientSocket`、`SSFObject_NetworkSocket`。

2) 协议层（Protocol Layer）
- 职责：封包、拆包、序列化、反序列化、版本兼容。
- 新增建议：`SSFModule_NetworkProtocol` 或工具类（先静态类）。

3) 会话层（Session Layer）
- 职责：连接身份管理、心跳、超时、重连 token、玩家绑定。
- 新增建议：`SSFModule_NetworkSession`（服务端优先）。

4) 消息分发层（Router Layer）
- 职责：按 `MsgId` 路由到业务处理器。
- 新增建议：`SSFModule_NetworkRouter`。

5) 业务层（Gameplay Layer）
- 职责：登录、角色、战斗、背包等业务消息处理。
- 建议：后续由各业务插件注册消息处理器到 Router。

## 4. 协议设计（MVP 版）

## 4.1 包头结构

建议统一包头（固定长度），示例：

- `Magic`：2 字节，协议魔数（防错包）
- `Version`：1 字节，协议版本
- `Flags`：1 字节（压缩/加密/ACK 等）
- `MsgId`：2 字节，消息类型
- `Seq`：4 字节，消息序号
- `BodyLen`：4 字节，负载长度
- `Checksum`：4 字节（可选）

说明：

- TCP 下也要做包边界定义，避免粘包/半包问题。
- `BodyLen` 必须有上限（如 64KB）防止恶意包。

## 4.2 消息类型建议（首批）

- `1001`：`C2S_Heartbeat`
- `1002`：`S2C_HeartbeatAck`
- `1101`：`C2S_Login`
- `1102`：`S2C_LoginAck`
- `1201`：`C2S_PlayerInput`
- `1202`：`S2C_PlayerState`
- `1301`：`S2C_Notify`

## 4.3 序列化方案

MVP 建议：

- 先用项目现有可控方式（结构体 + 二进制流）
- 第二阶段切换到 `Protobuf`（更易版本兼容）

理由：

- MVP 先上线、先打通链路。
- 数据模型稳定后再升级序列化协议，成本最低。

## 5. 会话与心跳

## 5.1 服务端 Session 结构（建议）

字段建议：

- `SessionId`
- `Socket`
- `LastRecvMs`
- `LastSendMs`
- `LastHeartbeatMs`
- `PlayerId`（登录后绑定）
- `State`（Connected/Auth/Playing/Closing）

## 5.2 心跳机制

建议值：

- 客户端每 `5s` 发一次心跳
- 服务端超时阈值 `15s`

规则：

- 超时则标记断线并触发清理。
- 心跳 ACK 可附带服务器时间，给客户端做时间校准。

---

## 6. 客户端重连机制（适配你当前实现）

当前客户端已具备基础重连逻辑，建议补齐以下策略：

- 固定间隔 + 抖动（如 3s ± 0.5s），避免重连风暴。
- 最大重试次数（可配置）与退避策略。
- 区分错误：
  - 连接拒绝（服务器未启动）
  - 网络不可达
  - 认证失败（无需重连）

重连后流程：

1. 建链成功
2. 发送重连凭证（Token + SessionKey）
3. 服务端恢复玩家状态或走重登流程

---

## 7. Tick 驱动下的处理循环（当前框架适配）

在 `SSFModule_NetworkServer::Tick` 中建议形成固定阶段：

1. `AcceptNewConnections()`
2. `RecvPackets()`（按 session 读）
3. `DecodeAndRoute()`
4. `UpdateSessions()`（心跳、超时、状态）
5. `FlushSendQueue()`

在 `SSFModule_NetworkClient::Tick` 中建议：

1. `ReconnectIfNeeded()`
2. `RecvPackets()`
3. `HandleServerMessages()`
4. `FlushSendQueue()`

好处：

- 时序稳定，便于定位 bug。
- 后续接入性能统计与压测更容易。

## 9. 技术难点与对应策略

1) 粘包半包
- 策略：固定包头 + `BodyLen` + 接收缓冲区状态机。

2) 线程安全（未来多线程）
- 策略：网络线程与逻辑线程消息队列解耦；当前可先单线程 Tick。

3) 延迟抖动
- 策略：客户端插值、预测、服务端关键帧纠偏。

4) 反作弊
- 策略：服务端权威判定；客户端只上传输入不上传最终结果。

5) 版本兼容
- 策略：包头带 `Version`，消息体字段采用向后兼容规则。

## 10. 分阶段落地计划

### 阶段 A（1~2 周，先可用）

- 完成统一包头
- 完成编码/解码
- 完成服务端 Router（按 MsgId 分发）
- 跑通心跳与基础登录消息

验收标准：

- 客户端稳定连接并周期心跳
- 服务端能正确识别/分发消息
- 断线可检测并清理

### 阶段 B（2~4 周，提升稳定性）

- 会话管理完善（Session 状态机）
- 客户端重连凭证机制
- 网络错误码细化与监控日志

验收标准：

- 断网恢复流程可用
- 异常日志可定位到消息与会话

### 阶段 C（按项目节奏，优化性能）

- 引入消息优先级队列
- 引入增量状态同步
- 评估 UDP/KCP 通道（战斗高频）

## 11. 最小可运行消息流（示例）

1. Client 建连成功
2. Client 发送 `C2S_Login`
3. Server 返回 `S2C_LoginAck`
4. Client 定时发送 `C2S_Heartbeat`
5. Server 返回 `S2C_HeartbeatAck`
6. Client 上报 `C2S_PlayerInput`
7. Server 广播 `S2C_PlayerState`

## 12. 与当前代码现状的对应说明

当前你们已经完成了以下关键基础：

- 客户端/服务端插件配置分离
- 网络模块按配置加载（并修复了短名/全名匹配问题）

---

## 13. 推荐结论

如果你希望快速进入可玩状态，建议采用：

- **当前 TCP + 命令驱动 + 心跳重连 + 关键状态同步**

这条路线最契合当前框架现状，改造成本低，且能平滑演进到高实时方案。

当你后续需要更高实时性（动作/FPS）时，再在此基础上叠加 `UDP/KCP` 通道，不必推倒重来。

## 17. 阶段C+完成项（当前）

本次在阶段C基础上继续完成了自动处置策略：

- 异常阈值自动断开
  - 当会话 `InvalidPacketCount + DroppedMsgCount` 超过阈值（默认 `10`）时，自动断开连接
- IP 黑名单冷却
  - 触发异常阈值后，将客户端 IP 加入黑名单（默认 `60s`）
  - 黑名单期间新连接在 `accept` 后立即拒绝
- 黑名单自动过期
  - 超过冷却时间后自动从黑名单移除
- 统计日志增强
  - 会话统计增加 `IP / Violations` 维度
  - 黑名单加入会打印原因与过期时间

说明：

- 当前实现是“轻量防护版”，适合先上线兜底。
- 后续可继续扩展成持久化封禁（写盘/DB）、按账号封禁、分级处罚策略。

### 17.1 ServerConfig 新增可配置项

为避免每次调参都改代码，阶段C/C+关键阈值已支持从 `ServerConfig.skywalkerC` 读取：

- `SessionTimeoutMS`：会话超时时间（毫秒）
- `MaxMsgPerSecond`：每会话每秒最大消息数
- `StatsLogIntervalMS`：异常统计日志输出间隔（毫秒）
- `ViolationThreshold`：触发自动断开+黑名单的异常阈值
- `BlacklistDurationMS`：黑名单冷却时长（毫秒）

配置示例：

```
ServerConfig
  IP=127.0.0.1
  Port=9527
  SessionTimeoutMS=15000
  MaxMsgPerSecond=120
  StatsLogIntervalMS=5000
  ViolationThreshold=10
  BlacklistDurationMS=60000
```

## 18. 阶段C++完成项（当前）

阶段C++已完成黑名单持久化能力，重启后不丢失封禁状态。

### 18.1 能力说明

- 启动加载：服务端启动时从黑名单文件加载未过期条目
- 变更落盘：新增拉黑、过期清理后会回写文件
- 自动过期：超过 `UntilMS` 的条目会自动失效并从内存/文件清理

### 18.2 新增配置项

- `BlacklistPersistence`：是否启用持久化（`1/true` 启用）
- `BlacklistFilePath`：黑名单文件路径（支持相对 `ServerConfig` 目录）

配置示例：

```
ServerConfig
  IP=127.0.0.1
  Port=9527
  SessionTimeoutMS=15000
  MaxMsgPerSecond=120
  StatsLogIntervalMS=5000
  ViolationThreshold=10
  BlacklistDurationMS=60000
  BlacklistPersistence=1
  BlacklistFilePath=ServerBlacklist.data
```

### 18.3 持久化文件格式

当前使用轻量文本格式，每行一条：

```
<IP> <UntilMS>
```

示例：

```
127.0.0.1 123456789000
192.168.1.10 123456790000
```

说明：

- `UntilMS` 为稳态毫秒时间戳，用于判定过期。
- 该格式后续可平滑迁移到 JSON/数据库。

```
  BlacklistHotReload=1
  BlacklistReloadIntervalMS=5000
```

### 19.4 运维验证步骤（手工）

建议按以下步骤验证热加载是否生效（无需重启服务）：

1. 确认 `ServerConfig.skywalkerC` 中已开启：
   - `BlacklistPersistence=1`
   - `BlacklistHotReload=1`
2. 启动服务端后，打开黑名单文件（默认 `ServerBlacklist.data`）。
3. 保留首行版本头，并追加一条未来过期时间记录，例如：

```
#Version 1
127.0.0.1 9999999999999
```

4. 保存文件，等待 `BlacklistReloadIntervalMS` 周期。
5. 从对应 IP 发起新连接，预期会被服务端立即拒绝。
6. 删除该行或改为已过期时间，再次保存并等待下个周期，预期连接恢复。

说明：

- 热加载以文件修改时间为触发条件，必须落盘保存。
- 若文件格式错误（非版本头/非法行），该行会被忽略并输出日志。
